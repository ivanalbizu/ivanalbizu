He dividido este bloque de "buenas prácticas" para tratar de mantener un hilo conductor, pero todo está muy relacionado

La idea es tratar de crear maquetaciones lo más optimizadas posibles:

  1. No nos llamen para decirnos que esta maquetación está mal o rompe por tal sitio
  2. Que sea mantenible y ampliable: que no sea un dolor de cabeza ampliar alguna maqueta para añadir nuevos cosas de usos

Buenas prácticas: Pasos previos

  • Soporte a navegadores
  • Analizar diseños / dudas
  • Dibujar en papel layout
  • Identificar componentes
  • HTML con reset de estilos
  • Estilos para tipografías y cajas
  • Reglas CSS

Pararse un poco a pensar antes de ponerse a maquetar como locos

  1. Navegadores: saber que versiones de navegadores tienen que ser soportados. Así sabremos que cosas podemos o no podemos usar. Esto puede venir definido por contrato, es buena idea preguntarlo y tratar de obtener información. Por ejemplo el aspect-ratio o gap para grid o flexbox.
  2. Diseños: en muchas ocasiones los diseños tienen contenidos "dummy" con texto e imágenes ideales. Es interesante plantear dudas de flujo de textos y comportamientos con diferentes tamaños de imágenes
  3. Complejidad: antes de empezar, es buena idea dibujar las cajas del layout o del componente
  4. Componentes: revisar diseños para localizar posibles componentes o elementos que puedan considerarse similares, que con pequeñas modificaciones se realice sus maquetas
  5. Generar HTML: añadir todo el HTML (de página global o componente) previamente a la maquetación y limpiar todos los estilos
  6. Estilos tipografías: definir las tipografías según diseños. Ojo que a veces los diseños no usan correctamente los line-height, a veces se usan para conseguir separaciones verticales con resto de elementos
  7. Estilos para cajas: primero espaciados interiores y luego exteriores
  8. Reglas CSS:
    • Usar selectores mediante clases o atributos. Evitar #ID
    • Evitar dar estilos directamente a etiquetas
    • Ser coherente con nombres para selectores: que representen lo que se pretende estilar. Sintáxis coherente: capitalize, guiones...

¿Qué es @container?

Las @container queries se asemejan a las @media queries.
Las @media queries hacen referencia al ancho de pantalla y las @container queries hacen referencia al ancho de un elemento que tengamos definido como contenedor.

En este ejemplo se ven 6 contenedores que tienen mismo contenido pero que tienen menos anchos unas que otras, usando los contenedores cambiamos la distribución de elementos, su visibilidad y valores de espaciados y tipografías

Más adelante veremos que actualmente existen dos variantes: una asociada a tamaño de contenedores y otra a valores de atributos. Y además, aparecen nuevas unidades CSS

Definición de @container | HTML
Recomiendo tener una caja sólo para definirlo como contenedor sin aplicar estilos, si acaso estilos estéticos (colores, bordes, ...) pero nada de flex ni grid y aplicar estilos a los hijos

Definición de @container | CSS

              
                [data-container="btn"] {
                  container-type: inline-size;// size | inline-size | normal
                  container-name: btns;
                  //container: <container-name> / <container-type>;
                }
                .btn-groups {
                  display: flex;
                  flex-direction: column;
                }
                @container btns (min-width: 300px) {
                  .btn-groups {
                    flex-direction: row;
                  }
                }
                @container (width > 900px) {
                  .card {
                    --bg-color: brown;
                    --padding: 16px;
                  }
                }
  1. Definición del elemento que será el contenedor
    • size: dimensiones block e inline eje. Aplica layout, style y size.
    • inline-size: dimensiones inline eje. Aplica layout, style y inline-size.
    • normal: no se trata como contenedor
  2. Nombre del contenedor
    • Opcional si sólo se tiene un contenedor
    • Siempre le doy nombre para evitar "cosas extrañas"
  1. Aplicamos estilos por defecto como a cualquier elemento
  1. Aplicamos estilos a cualquier elemento hijo del contenedor
  1. Podemos usar operadores <>

Unidades propias de @container

Han aparecido nuevas unidades CSS que aplican a @contenedores
  • cqw: % container width (vw)
  • cqh: % container height (vh)
  • cqi: % container inline size
  • cqb: % container block size
  • cqmin: menor de los valores cqi | cqb
  • cqmax: mayor de los valores cqi | cqb

Unidades propias de @container

              
                .title-card {
                  font-size: clamp(1rem, 3cqi + 1rem, 4rem);
                  line-height: 1.1;
                  margin-bottom: 1rem;
                }

Se puede aplicar a cualquier atributo que admita unidades. En este caso vemos ejemplo aplicado a tamaño de tipografía

CSS Painting API: Canvas I

            
              registerPaint('btn-pulse', class {
                static get contextOptions() { }

                static get inputArguments() { }

                static get inputProperties() { }

                _paintCircles(ctx, fillStyle, strokeStyle, x, y, rad)  { }

                paint(ctx, size, props) { }
              })
  1. paint() {}: método que se ejecuta cada vez que el motor CSS necesite pintar/repintar el elemento
  2. static get contextOptions() {}: método estático para habilitar/deshabilitar opacitdad de colores
  3. static get inputArguments() {}: método estático para exponer atributos que queramos usar en Paint.
  4. static get inputProperties() {}: método estático para acceder a Custom Properties CSS.
  5. _paintCircles() {}: método auxiliar para no repetir código.

Capas @layer
